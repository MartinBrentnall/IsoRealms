Suggestions for UI component framework design
I want to discuss the design of a framework for graphical UI components.

Although this doesn't typically apply directly to most game development, it can be relevant for someone who wants to implement a game that has editing facilities or a lot of user interface components (e.g. strategy games), so I will open it up to suggestions here.

The first thing to note is that I'm not really talking about usability or functional design; that's a very different area on which one could write an entire book.

I would like to discuss technical design and best practices for designing mechanisms for handling UI components (e.g. layouts, focusing, input mechanisms), etc. I'll make no secret of the fact that I intend to use this information in the IsoRealms engine.


Motivation for a UI framework

There are already lots of GUI frameworks out there; MFC, AWT, Swing, GTK, QT, Motif, Wxwidgets, etc, so why reinvent the wheel?

The problem with generic UI frameworks is that they run independently of the application "canvas" or "viewport". If you're making a game, you're probably going to want it to run fullscreen, and so you're going to want your UI components to be embedded within the frame and context of your application, rather than as separate components outside of it.

In addition, most generic frameworks mimic the look & feel of either the OS or something similar and end up looking out of place when used in a game. This is important because a typical OS GUI can easily break the atmosphere of your game, whereas a custom framework will provide you with a much higher level of both aesthetic and functional integration.


Layout and a brief look at Java Swing

I've been using Swing for several years now, and may have acquired a large influence from it in my own designs. Despite that, Swing has some aspects that I would like to change. One such aspect is the layout mechanism.

In Swing, a panel accepts an Object as a layout. My problem with this is that there isn't really anything useful you can do with an Object besides casting it to something that is useful, and I would like to avoid casting to as much extent as possible in my code, as I consider this bad practice.


A proposed layout solution

In my design, I have several interfaces and classes, which I will describe below:
  • Component
  • RectangularComponent
  • ComponentContainer

A Component is a simple interface for displaying and interacting with a UI component on screen. The component is drawn within (-1 to +1) on each axis, and events are passed as required to the component. Every aspect of the UI is a component, including the top level dialogues themselves.

Whereas a Component can be of any shape or size, a RectangularComponent is an abstract base class representing a component to occupy (but not necessarily fill) a specific rectangular area. The majority of typical components, such as buttons, text fields, lists, dialogue windows, etc. will extend this class. This class is analogous to Java's Component class, which must always have defined rectangular bounds.

The RectangularComponent leads to my proposed layout mechanism, for which a new interface is defined: ComponentBoundsCalculator. Implementations of this interface are responsible for calculating the bounds of a component (e.g. getLeft(), getRight(), getTop(), getBottom()). Each RectangularComponent will hold an instance of ComponentBoundsCalculator to calculate its bounds.

I may be overlooking something, but as far as I can tell, this simple mechanism should be able to handle any possible layout. For example, if you want a button to be placed to the right of another button and have the same height, you would implement ComponentBoundsCalculator as follows for the button:

Code:
class ButtonPosition:public ComponentBoundsCalculator {
  private:
  float spacing = 0.02f;
  Button* thisButton;
  Button* relatedButton;

  public:
  float getLeft() {
    return relatedButton->getRight() + spacing;
  }

  float getRight() {
    return getLeft() + thisButton->getWidth();
  }

  float getTop() {
    return relatedButton->getTop();
  }

  float getBottom() {
    return relatedButton->getBottom();
  }
};
This implementation would then be passed to button (thisButton) that you want placing to the right of the related button.


Problems with this approach

There are three problems with this approach: Overhead, impracticality and possible recursion. I will address each below.

Overhead: The overhead issue is due to the fact that functions are called each time we wish to retrieve the bounds of a component. This flies in the face of Java Swing, which simply returns a value. The problem is that there is no telling how long it will take to calculate the bounds of any particular component. If many components are "chained" together in terms of their relationship, it may become slow to calculate the bounds of the last members in the chain.

I believe this issue needs to be addressed with some sort of caching and change notification mechanism, but I won't concern myself too much with performance at the moment.

Practicality: If examples such as that above where applied to every component in an application, the sheer number of ComponentBoundsCalculator implementations would quickly become unweildly.

To solve this problem, it would be necessary to find ways of implementing ComponentBoundsCalculator in generic, reusable fashions, and reuse the same implementation class many times for different components. If done correctly, it should only be necessary to implement ComponentBoundsCalculator in special exceptional cases that the generic implementations do not cover. I consider this an essential requirement for the framework design.

Possible recursion: This problem arises when a layout is arranged in which component relation is recursive. Such layout would cause an infinite loop. A simple example would be specifying that two buttons should be laid out next to each other, i.e:
  • ButtonA should be placed to the left of ButtonB
  • ButtonB should be placed to the right of ButtonA

I don't really see any possibility for solving this in a generic design sense; I think it must be up to the client code to ensure that such recursions do not occur (any case of recursion I can think of wouldn't make much sense to attempt anyway, but I have a feeling it could easily be attempted by mistake). I am open to suggestions however. The caching proposed above may solve this issue to some extent.


Component input and focusing

I will confess right now that I have no idea how focusing works internally within Java Swing or any other API. What I propose is simply my own approach, from scratch.

The ComponentContainer represents a component that hierarchically contains other components. The top level window of the application itself must be a ComponentContainer.

In order to direct input events to the correct component, such events will be handled as follows:
  • Input event is recieved by the ComponentContainer
  • Input event is sent in top-to-bottom order to the child components. At this point, components do not process the event; they only test whether the event causes the component to gain focus, and return a true value if so.
  • If no component grabs the focus, the input event is tested by the ComponentContainer to see if it should change the focused component itself (this is to allow the equivalent of alt-tab'ing in Windows).
  • If a component grabs the focus, the focused component is changed in the ComponentContainer, otherwise it remains the same.
  • The input event is passed to the focused component for processing.

To me, this sounds very simple. Too simple. I'm sure I must be missing many things about proper focus management in regard to UI design. I've seen inside Swing and AWT and there is a lot of stuff in there! Any other suggestions and considerations would be very welcome.


Updating and animation

This is typically a very important aspect of good UI design. In my case however, the nature of my application will mean that it will be running in a continuous loop, repainting everything regardless at every frame anyway, so I haven't taken any time to think about it. I imagine this will be the case for many (most?) games.

If anyone wishes to comment or suggest anything about this, please feel free to do so.


Other consideration

This section is to cover those things I've forgot about. If anyone has anything else to say about the design of component UI frameworks, please feel free to do so! The more thoughts we can get, the more we can all benefit from the considerations that others may not have thought of!

Anyway, I'm glad to get all that out of my head and that's all I have to say on the matter!

Discuss!
